# Документация этапа 1: Концептуальное проектирование

Данная директория содержит документацию концептуального проектирования системы парсинга статей (Этап 1 согласно [DEVELOPMENT_PLAN.md](../DEVELOPMENT_PLAN.md)).

## 📚 Содержание документации

### [01. Концептуальный дизайн](01-conceptual-design.md)
**Основной документ**, описывающий концепцию системы:
- Анализ существующих решений (Scrapy, Beautiful Soup, Newspaper3k, Readability)
- Целевые веб-сайты и форматы (новостные порталы, блоги, научные публикации)
- Концептуальная архитектура системы
- Выбор технологического стека с обоснованием
- Ключевые архитектурные решения (модульность, расширяемость, асинхронность)
- Стратегия парсинга (многоуровневый подход)
- Модель данных (Article, Source, Image, ParsingTask)
- Безопасность и этика (robots.txt, rate limiting)
- Метрики и KPI
- Риски и митигация

### [02. Диаграммы вариантов использования](02-use-case-diagrams.md)
**Use Case диаграммы** для всех основных сценариев:
- 10 детальных use cases с последовательными диаграммами
- Акторы: Администратор, Пользователь, Разработчик, Планировщик
- Основные сценарии:
  - UC-01: Добавить источник
  - UC-02: Настроить правила парсинга
  - UC-03: Запустить парсинг вручную
  - UC-04: Просмотреть статьи
  - UC-05: Экспортировать данные
  - UC-06: Автоматический парсинг по расписанию
  - UC-07: Получить статью через API
  - UC-08: Мониторинг системы
  - UC-09: Управление очередью задач
  - UC-10: Обновить конфигурацию источника
- Диаграммы последовательностей (Sequence Diagrams) для каждого use case

### [03. Архитектурные диаграммы](03-architecture-diagrams.md)
**Визуализация архитектуры** на различных уровнях:
1. **Контекстная диаграмма** - система как целое и её окружение
2. **Контейнерная диаграмма** - основные приложения и хранилища
3. **Компонентная диаграмма** - внутренняя структура и модули
4. **Диаграмма развертывания** - физическое размещение в production
5. **Диаграмма потоков данных** - движение данных через систему
6. **Диаграмма взаимодействия** - runtime взаимодействия между модулями

Все диаграммы в формате **Mermaid** для легкой интеграции с Markdown.

### [04. Целевые сайты и форматы](04-target-websites-formats.md)
**Подробная спецификация** поддерживаемых источников:
- Категории целевых веб-сайтов:
  - Новостные порталы (РИА, ТАСС, Lenta.ru, Meduza)
  - Блог-платформы (Habr, VC.ru, Medium)
  - Персональные блоги (WordPress, Ghost, Jekyll)
  - Научные публикации (arXiv, технические блоги)
- Примеры конфигураций для конкретных сайтов (YAML)
- Форматы входных данных (HTML5, Open Graph, JSON-LD, RSS/Atom)
- Форматы выходных данных:
  - Схема PostgreSQL
  - JSON API формат
  - Экспорт: JSON, CSV, Markdown, HTML
- Стратегия определения формата (приоритетная последовательность)
- Валидация извлеченных данных
- Обработка специальных случаев (paywall, JavaScript, многостраничные статьи)

### [05. Технологический стек](05-technology-stack.md)
**Полная спецификация технологий** с обоснованием выбора:

**Core Stack**:
- Python 3.11+ (основной язык)
- httpx 0.27+ (HTTP клиент)
- Beautiful Soup 4 + lxml (парсинг HTML)
- PostgreSQL 15+ (база данных)
- SQLAlchemy 2.0+ (ORM)
- Redis 7+ (кэш и очередь)
- Celery 5.3+ (асинхронные задачи)
- FastAPI 0.109+ (REST API)
- Click 8.1+ (CLI)

**Supporting Stack**:
- Pydantic 2.5+ (валидация)
- pytest 7.4+ (тестирование)
- Ruff 0.1+ (линтинг и форматирование)
- mypy 1.8+ (проверка типов)
- structlog 24.1+ (логирование)
- prometheus-client 0.19+ (метрики)

**Infrastructure**:
- Docker + Docker Compose (контейнеризация)
- GitHub Actions (CI/CD)
- Prometheus + Grafana (мониторинг)
- Loki + Grafana (логи)

Для каждой технологии приведены:
- Обоснование выбора
- Примеры использования
- Сравнение с альтернативами
- Конфигурация

## 🔄 Связь с MBSE

Документация следует принципам **Model-Based Systems Engineering (MBSE)**:

1. **Requirements Engineering**: Определены функциональные и нефункциональные требования
2. **System Architecture**: Разработана модульная архитектура с четкими интерфейсами
3. **Component Design**: Каждый компонент спроектирован с определением ответственности
4. **Use Cases**: Описаны сценарии использования системы
5. **Technology Selection**: Обоснованный выбор технологий
