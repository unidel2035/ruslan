# Концептуальное проектирование системы парсинга статей

## Обзор

Данный документ описывает концептуальную архитектуру системы автоматического парсинга статей с веб-сайтов. Документ создан в рамках этапа 1 разработки согласно DEVELOPMENT_PLAN.md и определяет ключевые концепции, подходы и архитектурные решения.

## 1. Анализ существующих решений

### 1.1 Обзор популярных библиотек и фреймворков

#### Scrapy
**Описание**: Полнофункциональный фреймворк для масштабного краулинга и извлечения данных.

**Преимущества**:
- Высокая производительность за счет асинхронной архитектуры
- Встроенные механизмы управления запросами, кэширования
- Расширяемость через middleware и pipelines
- Поддержка распределенного краулинга (Scrapy-Redis)

**Недостатки**:
- Высокий порог входа для простых задач
- Избыточность для небольших проектов

#### Beautiful Soup
**Описание**: Легковесная библиотека для парсинга HTML/XML.

**Преимущества**:
- Простота использования
- Гибкие селекторы (CSS, атрибуты)
- Хорошо работает с различными парсерами (lxml, html.parser)

**Недостатки**:
- Отсутствие встроенных механизмов загрузки страниц
- Требует дополнительных библиотек для полноценного решения
- Низкая производительность на больших объемах данных

#### Newspaper3k
**Описание**: Специализированная библиотека для извлечения статей из новостных сайтов.

**Преимущества**:
- Автоматическое определение основного контента статьи
- Извлечение метаданных (автор, дата, изображения)
- Поддержка 38 языков
- Встроенная обработка NLP

**Недостатки**:
- Ограниченная гибкость для нестандартных сайтов
- Зависимость от эвристик, которые могут не работать для всех сайтов

#### Readability Algorithm
**Описание**: Алгоритм извлечения основного контента, используемый в Firefox Reader View.

**Принцип работы**:
1. Парсинг HTML в DOM-структуру
2. Оценка каждого элемента на основе:
   - Типа тега (p, div, article и т.д.)
   - Класса и ID
   - Плотности ссылок (link density)
   - Длины текста
3. Выбор элемента с наивысшей оценкой
4. Удаление "мусора" (реклама, меню, футер)

**Преимущества**:
- Работает без предварительной конфигурации
- Хорошо подходит для большинства новостных сайтов
- Открытая реализация на нескольких языках

**Недостатки**:
- Может ошибаться на нестандартных макетах
- Требует тонкой настройки для специфических сайтов

### 1.2 Рекомендуемый подход

**Гибридная архитектура**, комбинирующая:
1. **Scrapy** - как основа для краулинга и управления запросами
2. **Beautiful Soup + lxml** - для гибкого парсинга HTML
3. **Readability-подобный алгоритм** - для автоматического извлечения контента
4. **Конфигурируемые селекторы** - для специфических сайтов

## 2. Целевые веб-сайты и форматы

### 2.1 Категории целевых сайтов

#### Новостные порталы
- **Примеры**: lenta.ru, meduza.io, ria.ru, tass.ru
- **Особенности**: Стандартизированная структура, Open Graph теги, schema.org разметка
- **Форматы**: HTML5 с семантическими тегами (article, header, time)

#### Блог-платформы
- **Примеры**: Medium, Habr, vc.ru, dzen.ru
- **Особенности**: Единая структура для всех статей платформы
- **Форматы**: HTML + JSON-LD для метаданных

#### Персональные блоги
- **Примеры**: WordPress, Ghost, Jekyll блоги
- **Особенности**: Вариативная структура, зависит от темы
- **Форматы**: Разнообразные HTML-структуры

#### Научные публикации
- **Примеры**: arxiv.org, habr.com/ru/articles/, medium.com technical blogs
- **Особенности**: Сложная структура с формулами, кодом, диаграммами
- **Форматы**: HTML с специальными элементами (MathJax, код-блоки)

### 2.2 Поддерживаемые форматы данных

#### Входные форматы
- HTML5 с семантической разметкой
- HTML4/XHTML
- Метаданные: Open Graph, Twitter Cards, schema.org (JSON-LD, Microdata)
- RSS/Atom feeds (для обнаружения статей)

#### Выходные форматы
- JSON (структурированные данные)
- Markdown (для текстового контента)
- HTML (очищенный контент)
- База данных (PostgreSQL/MongoDB схема)

### 2.3 Извлекаемые данные

**Обязательные поля**:
- Заголовок статьи
- Основной текст
- URL источника
- Дата парсинга

**Опциональные поля**:
- Автор(ы)
- Дата публикации
- Дата обновления
- Изображения (главное изображение, изображения в тексте)
- Теги/категории
- Краткое описание (summary)
- Язык контента
- Количество просмотров/лайков (если доступно)
- Связанные статьи

### 3.2 Описание компонентов

#### Scheduler (Планировщик)
**Назначение**: Планирование периодических задач парсинга

**Функции**:
- Создание задач по расписанию
- Управление приоритетами
- Обработка повторных попыток

**Технологии**: Celery Beat

#### Task Queue (Очередь задач)
**Назначение**: Асинхронная обработка задач парсинга

**Функции**:
- Распределение задач между воркерами
- Мониторинг статуса выполнения
- Обработка сбоев

**Технологии**: Celery + Redis/RabbitMQ

#### Crawler/Fetcher (Краулер)
**Назначение**: Загрузка веб-страниц

**Функции**:
- HTTP/HTTPS запросы
- Обработка куки и сессий
- Управление user-agent
- Rate limiting (ограничение частоты запросов)
- Retry механизм
- Прокси-поддержка

**Технологии**: httpx (async), aiohttp

#### Content Parser (Парсер контента)
**Назначение**: Извлечение структурированных данных из HTML

**Функции**:
- Парсинг HTML в DOM
- Применение селекторов (CSS, XPath)
- Извлечение метаданных (Open Graph, schema.org)
- Автоматическое определение контента (Readability)
- Применение правил для конкретных сайтов

**Технологии**: Beautiful Soup 4, lxml, custom readability implementation

#### Data Processor (Обработчик данных)
**Назначение**: Очистка и нормализация извлеченных данных

**Функции**:
- Удаление HTML-тегов из текста
- Нормализация пробелов и переносов
- Валидация данных
- Обогащение данных (NLP, извлечение сущностей)

**Технологии**: bleach, html2text, custom validators

#### Storage Layer (Слой хранения)
**Назначение**: Сохранение извлеченных статей

**Функции**:
- Сохранение в базу данных
- Кэширование часто запрашиваемых данных
- Версионирование статей
- Дедупликация

**Технологии**: PostgreSQL, SQLAlchemy, Redis

#### Configuration Manager (Менеджер конфигурации)
**Назначение**: Управление конфигурациями источников

**Функции**:
- Загрузка конфигураций из YAML/JSON
- Валидация конфигураций
- Горячая перезагрузка правил

**Технологии**: Pydantic, YAML

#### API Layer (API слой)
**Назначение**: Предоставление интерфейса для взаимодействия

**Функции**:
- REST API для CRUD операций
- WebSocket для real-time обновлений
- Аутентификация и авторизация
- Rate limiting

**Технологии**: FastAPI, WebSockets

#### CLI Interface (CLI интерфейс)
**Назначение**: Интерфейс командной строки

**Функции**:
- Запуск парсинга из командной строки
- Управление конфигурациями
- Просмотр статистики
- Экспорт данных

**Технологии**: Click или Typer

## 4. Выбор технологического стека

### 4.1 Основной язык программирования

**Выбор: Python 3.11+**

**Обоснование**:
- Богатая экосистема библиотек для веб-скрапинга
- Простота разработки и поддержки
- Хорошая производительность с async/await
- Широкое сообщество и документация

### 4.2 Библиотеки и фреймворки

#### HTTP-клиент
**Выбор: httpx**

**Обоснование**:
- Поддержка sync и async API
- HTTP/2 поддержка
- Современный, активно развивающийся
- Совместимость с requests API

**Альтернатива**: aiohttp (если нужна только async)

#### HTML-парсер
**Выбор: Beautiful Soup 4 + lxml**

**Обоснование**:
- Простота использования
- lxml - самый быстрый парсер
- Гибкие селекторы
- Устойчивость к невалидному HTML

#### База данных
**Выбор: PostgreSQL**

**Обоснование**:
- Надежность и ACID-гарантии
- Мощные возможности индексации
- Полнотекстовый поиск (для поиска по статьям)
- JSON-поддержка (для метаданных)
- Поддержка расширений (pg_trgm для fuzzy search)

**Альтернатива**: MongoDB (если нужна большая гибкость схемы)

#### ORM
**Выбор: SQLAlchemy 2.0+**

**Обоснование**:
- Зрелое решение с большим сообществом
- Поддержка async (SQLAlchemy 2.0+)
- Мощный query builder
- Поддержка миграций через Alembic

#### Система очередей
**Выбор: Celery + Redis**

**Обоснование**:
- Celery - стандарт для асинхронных задач в Python
- Redis - быстрый и надежный брокер сообщений
- Поддержка периодических задач (Celery Beat)
- Мониторинг через Flower

**Альтернатива**: RQ (Redis Queue) - более простая альтернатива

#### API Framework
**Выбор: FastAPI**

**Обоснование**:
- Современный, быстрый фреймворк
- Автоматическая генерация OpenAPI документации
- Встроенная валидация через Pydantic
- Поддержка async
- WebSocket поддержка

#### CLI Framework
**Выбор: Click**

**Обоснование**:
- Простота создания CLI
- Хорошая документация
- Автоматическая генерация help
- Композиция команд

**Альтернатива**: Typer (более современная обертка над Click с type hints)

#### Тестирование
**Выбор: pytest**

**Обоснование**:
- Стандарт тестирования в Python
- Богатая экосистема плагинов
- Простота написания тестов

## 5. Ключевые архитектурные решения

### 5.1 Модульность

**Принцип**: Каждый компонент системы должен быть независимым модулем с четко определенным интерфейсом.

**Реализация**:
- Использование абстрактных базовых классов (ABC)
- Dependency Injection для связи компонентов
- Разделение на отдельные Python пакеты

**Преимущества**:
- Простота тестирования
- Возможность замены компонентов
- Параллельная разработка

### 5.2 Расширяемость

**Принцип**: Система должна легко расширяться новыми источниками и функциями.

**Реализация**:
- Plugin система для парсеров
- Конфигурационные файлы для источников
- Hook система для кастомной обработки

**Преимущества**:
- Добавление новых сайтов без изменения кода
- Кастомизация поведения через конфигурацию

### 5.3 Асинхронность

**Принцип**: Максимальное использование асинхронного I/O для повышения производительности.

**Реализация**:
- async/await для HTTP-запросов
- Celery для фоновых задач
- Redis для кэширования и очередей

**Преимущества**:
- Высокая производительность
- Эффективное использование ресурсов
- Масштабируемость

### 5.4 Отказоустойчивость

**Принцип**: Система должна gracefully обрабатывать ошибки и продолжать работу.

**Реализация**:
- Retry механизмы с exponential backoff
- Circuit breaker pattern для проблемных источников
- Логирование всех ошибок
- Мониторинг и алертинг

**Преимущества**:
- Надежность
- Простота диагностики проблем
- Минимизация downtime

### 5.5 Конфигурируемость

**Принцип**: Поведение системы должно настраиваться без изменения кода.

**Реализация**:
- YAML/JSON конфигурации для источников
- Environment variables для настроек окружения
- Pydantic модели для валидации конфигураций

**Преимущества**:
- Гибкость
- Простота добавления новых источников
- Возможность A/B тестирования правил

### 5.6 Наблюдаемость (Observability)

**Принцип**: Система должна предоставлять полную видимость своего состояния.

**Реализация**:
- Структурированное логирование
- Метрики (количество запросов, ошибки, латентность)
- Трейсинг распределенных транзакций
- Health checks

**Преимущества**:
- Простота отладки
- Проактивное обнаружение проблем
- Оптимизация производительности

## 6. Стратегия парсинга

### 6.1 Многоуровневый подход

**Уровень 1: Метаданные**
- Проверка Open Graph тегов
- Проверка Twitter Cards
- Проверка schema.org (JSON-LD, Microdata)

**Уровень 2: Конфигурированные селекторы**
- Применение правил для известных сайтов
- CSS/XPath селекторы из конфигурации

**Уровень 3: Readability алгоритм**
- Автоматическое определение основного контента
- Используется как fallback для неизвестных сайтов

**Уровень 4: Эвристики**
- Анализ структуры DOM
- Поиск характерных паттернов (article tags, date patterns)

### 6.2 Обработка ошибок

**Типы ошибок**:
1. **Network errors** (таймаут, DNS, соединение)
   - Retry с exponential backoff
   - Максимум 3 попытки

2. **HTTP errors** (404, 403, 500)
   - 404 - пометить источник как недоступный
   - 403/429 - применить rate limiting
   - 500 - retry позже

3. **Parsing errors** (невалидный HTML, не найден контент)
   - Логирование для анализа
   - Попытка fallback методов
   - Пометка для ручной проверки

4. **Validation errors** (неполные данные)
   - Сохранение частичных данных
   - Пометка для дополнения

### 6.3 Rate Limiting

**Стратегия**:
- Глобальный лимит: 100 запросов/минуту
- Лимит на домен: 10 запросов/минуту (настраиваемый)
- Adaptive rate limiting на основе HTTP headers (Retry-After)

**Реализация**:
- Redis для хранения счетчиков
- Token bucket algorithm

## 9. Метрики и KPI

### 9.1 Функциональные метрики

- **Покрытие источников**: Количество успешно парсящихся источников
- **Полнота данных**: % статей с полными метаданными
- **Точность парсинга**: % корректно извлеченного контента (требует ручной проверки выборки)

### 9.2 Производительность

- **Throughput**: Статей в час на один воркер (цель: >100)
- **Latency**: Среднее время парсинга одной статьи (цель: <5 сек)
- **Concurrent requests**: Количество одновременных запросов (цель: >50)

### 9.3 Надежность

- **Success rate**: % успешных парсингов (цель: >95%)
- **Error rate**: % ошибок по типам
- **Uptime**: Доступность системы (цель: >99%)
- **MTTR**: Среднее время восстановления после ошибки

### 9.4 Качество кода

- **Test coverage**: Покрытие тестами (цель: >80%)
- **Code quality**: Оценка по инструментам (ruff, mypy)
- **Documentation**: % задокументированных функций (цель: 100% публичных API)
